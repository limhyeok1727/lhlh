# CLAUDE.md

이 파일은 이 저장소에서 작업하는 Claude Code(claude.ai/code)에게 제공되는 가이드입니다.

## 프로젝트 개요

- 웹 액세스 로그를 요약해서 일일 트래픽 리포트(텍스트)를 만드는 작은 2단계 파이프라인입니다.
- 기술 스택: Python 3.9+ (`log_parser.py`가 `str.removesuffix`를 사용하므로 3.9+ 필요), 순수 표준 라이브러리만 사용. 빌드 시스템, 패키지 매니페스트가 없습니다.

## 명령어

- 로그 파일 요약: `python src/log_parser.py data/access_2026-07-15.log`
- 리포트 생성 (현재는 파싱된 로그 데이터가 아니라 하드코딩된 샘플 dict를 렌더링함): `python src/report.py`
- 테스트: `python3 -m pytest` (커밋 전 반드시 실행하고 결과를 보여줄 것 — 아래 Git 규칙 참고)
- 린트/빌드 도구는 별도로 설정되어 있지 않습니다.

## 아키텍처

두 스크립트는 암묵적인 dict 스키마로만 연결되어 있고, 그 사이에 공유 모듈/타입/검증 로직은 없습니다.

1. **`src/log_parser.py`** — `parse_line()`은 공백으로 구분된 로그 한 줄(`DATE TIME METHOD PATH STATUS LATENCYms`)을 인덱스 위치 기준으로 분리합니다. `summarize()`는 로그 파일을 한 줄씩 읽어 상태 코드를 2xx/3xx/4xx/5xx로 분류하고, 레이턴시를 누적하고, `Counter`로 엔드포인트별 요청 수를 셉니다. 공백 기준으로 6개 미만 필드인 줄은 조용히 건너뜁니다. 반환값은 `{total, 2xx, 3xx, 4xx, 5xx, avg_latency_ms, top_endpoints}` 형태의 통계 dict입니다.
2. **`src/report.py`** — 위와 동일한 통계 dict를 받아 `f1()`(헤더 + 상태별 건수/퍼센트), `f2()`(알림 규칙: 5xx 비율 5% 이상, 평균 레이턴시 300ms 이상), `f3()`(상위 엔드포인트)으로 텍스트를 렌더링하고 `make()`가 이를 조합합니다. `__main__` 블록은 `log_parser`를 호출하지 않고 하드코딩된 샘플 dict를 렌더링하므로, 둘을 실제로 연결하려면(예: `make(summarize(path))`) 현재는 수동으로 연결해야 합니다.

dict 스키마가 암묵적이고 검증되지 않기 때문에, 한쪽 파일의 키를 바꾸면 파싱/임포트 시점이 아니라 런타임에만(`KeyError`) 다른 쪽이 깨집니다. 통계 데이터의 형태(shape)를 바꿀 땐 두 파일을 함께 확인하세요.

## 코드 컨벤션

- 별도의 린터/포매터가 설정되어 있지 않으므로 기존 파일의 스타일(들여쓰기, 네이밍 등)을 따를 것
- `log_parser.py`와 `report.py`는 암묵적 dict 스키마로만 연결되므로, 통계 데이터의 형태(shape)를 바꿀 때는 두 파일을 함께 확인할 것

## Git 규칙

- 커밋 메시지는 "type: 요약" 형식(feat/fix/refactor/test/docs/chore)을 사용할 것. 본문에는 변경 이유를 적을 것.
- 커밋 전 반드시 `python3 -m pytest`를 실행하고 결과를 보여줄 것(MUST)
- main 브랜치에 직접 커밋하지 않음 — 작업은 브랜치에서 진행

## 금지 사항

- `data/` 폴더의 원본 로그 파일 수정 금지 (재현·테스트가 필요하면 사본을 만들어 사용)
- 사전 합의 없는 의존성 추가 금지 — 표준 라이브러리만 사용
- 명시적으로 동작 변경을 요청받지 않는 한 기존 동작(출력, 포맷, 함수 시그니처) 변경 금지 — 리팩터링, 스타일 정리, 버그 수정은 수정 대상 범위 밖의 동작을 바꿔서는 안 됨

## 위험 작업 확인

- 파일 삭제, 대량 이동/덮어쓰기, git push, 외부 전송은 실행 전에 반드시 확인을 받을 것(MUST)

## 다단계 작업 관리

- 3단계 이상의 작업은 실행 전에 계획을 먼저 보여주고 승인을 받을 것
- 진행 상태는 저장소 루트의 `PROGRESS.md`에 기록하고, 단계가 끝날 때마다 갱신할 것 (완료한 것 / 남은 것)

## 소통 방식

- 코드 자체(코드, 식별자, 파일 경로 등)를 제외하고는 한글로 답변할 것
- 모르는 것은 모른다고 말하고, 추측으로 진행하지 않을 것
- 에러가 나면 원인과 해결책을 함께 보고할 것

## 알려진 취약 지점

- `report.py`의 `f1()`/`f2()`는 0 여부 확인 없이 `d["total"]`로 나눕니다. 유효한 줄이 하나도 없는 로그가 들어오면 `log_parser.summarize()`가 `total: 0`을 반환하고, 이로 인해 `report.py`가 `ZeroDivisionError`로 크래시합니다 (`log_parser.py` 자체의 `total_latency // count`도 상류에서 동일한 문제를 갖고 있음).
